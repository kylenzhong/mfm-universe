## Top 3 buildable opportunities

### 1. A day-zero "should I swap?" harness for new open-weight drops
- **Category / stage:** Developer infrastructure / nascent — and newly urgent: the open frontier is now shipping multiple releases a week.
- **Velocity / cross-validation:** GLM-5.2 topped the like board (2,862) and its GGUF hit 164k dl *the same day it dropped*; DeepSeek-V4-Pro published hours ago; Kimi is #1 on Google. Meanwhile `ollama`'s own description still advertises **Kimi-K2.6 / GLM-5.1** — last cycle's weights — proof that even the default runtime can't keep up with the release cadence. The question "a new frontier open model just dropped; is it actually better for *my* lane, and should I switch?" is now a multiple-times-a-week event with no tooling around it.
- **Hypothesis:** Builders running local/open models hard-code one base on vibes and re-evaluate approximately never, because re-benchmarking is a manual afternoon each time. So they either chase every shiny drop blindly or ossify on a stale model. A harness that, on any new top-of-board weight, auto-runs *your* task suite and emits a swap/no-swap verdict with a pass-rate delta turns a recurring afternoon into a notification.
- **MVP (≤1 week):** CLI that watches the Hub like/download board, detects a new frontier-class drop (GLM-5.2 today), pulls it via ollama, runs a user-supplied eval set against the incumbent, and prints `SWAP` / `HOLD` + per-lane pass-rate delta. One demo: GLM-5.2 vs the user's current base.
- **Why now:** three open frontier weights (GLM-5.2, DeepSeek-V4-Pro, Kimi) surfaced within a day and the default runtime is two versions behind. The cadence broke human re-evaluation this week specifically.
- **Confidence: 4/5.**

### 2. A "capability-clone" kit — frontier agent + one task class → a small local model that matches it on that lane
- **Category / stage:** Developer infrastructure / nascent. Carryover from last week, and the loop tightened again over the weekend.
- **Velocity / cross-validation:** both ends of the produce→consume loop kept climbing. Inputs: `Glint-Research/Fable-5-traces` is now **461 likes / 35,512 dl** (up from 414 / 26,233 Friday), with `Crownelius/Complete-FABLE.5-traces-2M` and `armand0e/claude-fable-5-claude-code` (242 / 11,721) filling in behind it. Outputs: `yuxinlu1/gemma-4-12B-coder-fable5-composer2.5-v1-GGUF` is at **2,487 likes / 561,577 dl** and its `agentic-fable5-composer2.5` sibling at 241,409 — small models explicitly trained on frontier traces, out-pulling most base weights. The connecting pipeline between trace-set and trace-trained-model is still hand-rolled.
- **Hypothesis:** Teams want a cheap local model that's genuinely good at *their* one workflow. Today they pay frontier per-token forever or hand-build a fragile distillation run. A kit that drives a frontier model over N tasks of one lane, logs traces in the proven `Fable-5-traces` schema, fine-tunes a `gemma-4-12B` LoRA, and benchmarks lane pass-rate makes "clone this capability" one command.
- **MVP (≤1 week):** CLI that (a) drives a frontier model over a user task set, (b) emits Fable-schema traces, (c) fires a gemma-4-12B LoRA fine-tune, (d) prints a before/after lane benchmark. One screen-recorded clone of a single coding-agent behavior.
- **Why now:** the trace-datasets and the trace-trained GGUFs are *both* still climbing the board a week later — this isn't a one-day spike, it's a compounding loop, and the tooling between the halves is still artisanal.
- **Confidence: 4/5.**

### 3. A lane-specialist model registry — "the best local model for *this* task, this week"
- **Category / stage:** Developer tooling / nascent. The consumption side of #2's production side, sharpened by #1's release cadence.
- **Velocity / cross-validation:** the board is fragmenting into purpose-built fine-tunes out-downloading shinier bases: `gemma-4-12B-coder-fable5-composer` (561k dl) for code, `agentic-fable5-composer` (241k) for tool-use, `baidu/Unlimited-OCR` (1,289 likes / 363k dl) for OCR, `empero-ai/Qwythos-9B-Claude-Mythos` (907k dl) for roleplay, `deepreinforce-ai/Ornith-1.0` (9B + 35B GGUFs) for reasoning. No two lanes share a winner, and the winners rotate weekly. Guest fit: **Pieter Levels (0.88 local LLM)** — cost-aware, offline, exactly this user.
- **Hypothesis:** A builder picking a local model hard-codes one base and never revisits, leaving a 2×-better specialist on the table. A registry that tags the live Hub by task lane, ranks the freshest specialist GGUF per lane by download-velocity + likes, and emits a one-line `ollama pull` per lane is pure leverage on churn you don't track yourself.
- **MVP (≤1 week):** script that classifies the top ~50 trending GGUFs into ~8 task lanes, ranks each by download-velocity + likes, outputs a daily-refreshing one-line `ollama pull` per lane.
- **Why now:** specialist fine-tunes are out-downloading the base layer *and* a fresh frontier base (GLM-5.2) just reset the field — "which local model for my one task, today" became a real recurring question this week.
- **Confidence: 3/5.**
